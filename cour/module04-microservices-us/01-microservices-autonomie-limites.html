De « ça marche » à « c'est cadré »

Au module 03, vous avez fait tourner le service catalogue de SemiShop avec un agent IDE, et un front React l'affiche. La fonctionnalité existe, le diff est maîtrisé. Mais une application qui marche n'est pas une application cadrée : personne n'a écrit noir sur blanc ce que la boutique doit faire, pour qui, ni comment on saura que c'est fini. Personne n'a non plus décidé combien de services la composeront, ni dans quels langages. C'est le travail de cette journée.

Le fil conducteur tient en une tension : l'IA sait produire beaucoup, vite. Demandez-lui de « découper SemiShop en microservices » et elle en proposera dix, avec un bus d'événements et un service discovery. Votre métier, dans ce module, n'est pas de produire mais de cadrer et filtrer : garder ce qui a de la valeur, refuser la complexité gratuite.


1. Le microservice : ce que c'est, à quoi ça sert, un exemple

Ce que c'est. Un microservice est un composant applicatif autonome : il porte une responsabilité métier unique, expose une API stable, gère sa propre persistance et se déploie et s'observe indépendamment des autres. On l'oppose au monolithe, où toutes les responsabilités vivent dans un seul déployable.

À quoi ça sert. Découper permet à des équipes différentes de livrer à des rythmes différents, de mettre à l'échelle la partie qui sature, ou d'isoler une technologie particulière. C'est un moyen, jamais un objectif.

Un exemple simple : le socle SemiShop. La boutique de matériel reconditionné de SEMIFIR repose sur trois services :

flowchart LR
    t_front["Front React SPA"] -->|"HTTP /api"| t_bff["BFF exposition<br/>Spring Boot"]
    t_bff -->|"REST"| t_cmd["Service commandes<br/>Spring Boot"]
    t_bff -->|"REST"| t_cat["Service catalogue<br/>Spring Boot"]
    t_cmd -->|"REST consultation"| t_cat
    t_cmd -->|"SQL"| t_dbm[("metier_db")]
    t_cat -->|"SQL"| t_dbr[("referentiel_db")]

Le catalogue tient le référentiel produits (grades, prix, stock) dans referentiel_db. Le service commandes porte les règles métier (panier, statuts, paiement simulé) dans metier_db. Le BFF est la porte d'entrée du front : il agrège et authentifie. Trois responsabilités, trois services, deux bases logiques sur une même instance PostgreSQL.

🔷 Note : ce découpage minimal est une décision, pas une évidence. Panier, clients et paiements vivent aujourd'hui dans le service commandes ; le stock vit dans le catalogue. Faut-il les extraire ? C'est exactement la question du TD 4.1.

Les cinq attributs qui qualifient un microservice :

Attribut Ce que ça signifie Sur SemiShop
Autonomie Se déploie et évolue seul Le catalogue livre une correction de prix sans toucher aux commandes
Responsabilité métier Un domaine, un seul « Tenir le référentiel produits reconditionnés »
API stable Contrat explicite (REST, OpenAPI) GET /api/catalogue/produits, versionné
Persistance dédiée Sa base, jamais partagée en écriture referentiel_db vs metier_db
Observabilité Logs, métriques, traces propres Traces corrélées BFF → métier → référentiel

Pour les profils techniques : « persistance dédiée » ne signifie pas une instance PostgreSQL par service. Une instance unique avec une base logique par service suffit en développement — c'est le choix du lab. L'interdit porte sur l'accès croisé : un service ne lit jamais la base d'un autre, il passe par son API.

2. Les limites : chaque frontière a un coût

C'est le message central du module. Une frontière de service introduit un appel réseau (latence, pannes partielles), une cohérence distribuée à gérer (plus de transaction unique qui embrasse tout), une duplication d'infrastructure (image Docker, healthcheck, secrets, monitoring par service) et une complexité de test (orchestrer plusieurs services pour un scénario de bout en bout).

Pour les profils techniques : passer d'un appel de méthode in-process (nanosecondes, transactionnel, typé au compilateur) à un appel HTTP entre conteneurs (millisecondes, faillible, sérialisé) multiplie les modes de défaillance. Un NullPointerException devient un timeout, un 503, une double écriture. Sur SemiShop, la validation d'une commande décrémente le stock du catalogue : dans le socle, c'est un appel REST qui peut échouer après la création de la commande — un cas que le monolithe n'avait pas.

Prenez un exemple concret du domaine : le décrément de stock à la validation d'une commande. Tant que stock et commande vivent dans le même déployable, une transaction SQL garantit la cohérence. Séparez-les, et il faut choisir : appel synchrone avec compensation en cas d'échec, ou accepter une fenêtre d'incohérence. Aucune des deux options n'est gratuite.

🟨 Warning — le microservice artificiel : découper « parce que c'est moderne » est l'anti-pattern classique de ce module. Pour une boutique tenue par une équipe, un monolithe modulaire (modules bien séparés dans un seul déployable) ou un socle de deux-trois services est presque toujours le bon point de départ. On ne trace une frontière physique que lorsqu'un besoin réel l'exige.

3. La règle de décision : bénéfice > coût d'intégration

On découpe quand le bénéfice attendu dépasse le coût d'intégration, pas avant. Les signaux qui justifient une extraction (au moins un requis) :

Signal Ce qu'il veut dire Exemple SemiShop
Équipes autonomes Deux équipes veulent livrer sans se bloquer Une équipe dédiée au panier, une autre au catalogue
Scaling ciblé Une partie sature indépendamment La consultation catalogue encaisse dix fois le trafic du paiement
Cycle de vie divergent Un composant change beaucoup plus souvent Les règles de pricing du reconditionné bougent chaque semaine
Conformité / isolation Une contrainte impose la séparation Les données de paiement cloisonnées
Technologie imposée Un écosystème est nettement meilleur ailleurs Un scoring d'état du matériel en Python

Positionnons quelques candidats SemiShop sur l'arbitrage coût / bénéfice :

    title Decouper SemiShop - cout integration vs benefice
    x-axis Faible cout integration --> Fort cout integration
    y-axis Faible benefice --> Fort benefice
    quadrant-1 A etudier serieusement
    quadrant-2 Decoupage justifie
    quadrant-3 Rester module
    quadrant-4 Microservice artificiel
    Service panier equipe dediee: [0.55, 0.75]
    Service grade produit seul: [0.7, 0.2]

Le quadrant bas-droite — fort coût, faible bénéfice — est celui du microservice artificiel : un « service grade produit » qui ne ferait que stocker A/B/C n'a aucune raison d'exister hors du catalogue. À l'inverse, le catalogue isolé est peu coûteux (frontière naturelle, données de référence) pour un bénéfice réel.

🔹 Bonne pratique production : commencez par des frontières logiques propres (packages, modules par domaine), et n'extrayez un service que le jour où un signal concret le réclame. Extraire un module bien isolé est un chantier maîtrisé ; recoller trois microservices mal découpés est une réécriture.

4. Et l'IA dans tout ça ?

L'IA intervient deux fois dans cette décision, et jamais comme décideur. En amont, c'est un générateur d'arguments : demandez-lui les bénéfices et les coûts d'une extraction donnée, exigez qu'elle cite les signaux, interdisez-lui de supposer des exigences que vous n'avez pas fournies. En aval, c'est un rédacteur : une fois la décision prise, elle met en forme l'entrée du journal de décisions plus vite que vous.

Entre les deux, le filtre reste humain — c'est le triptyque du module 02 : proposition IA → gate déterministe → validation humaine. Ici, le gate est la grille de signaux : une extraction sans signal réel cité est refusée, quelle que soit l'éloquence de la justification générée.

Ce qu'il faut retenir

  • 🔷 Un microservice = autonomie + responsabilité métier + API + persistance dédiée + observabilité. Cinq attributs, pas un.
  • 🔷 On découpe quand le bénéfice dépasse le coût d'intégration. Chaque frontière ajoute réseau, cohérence distribuée, infra dupliquée et tests plus lourds.
  • 🔷 Le socle SemiShop (BFF + commandes + catalogue) est un découpage minimal décidé, pas un dogme : panier, stock, clients et paiement y vivent en modules.
  • 🔷 L'IA propose des découpages et rédige les décisions ; elle ne tranche jamais. Toute frontière doit citer un signal réel.

Pièges courants

🟨 Découper par réflexe. « Microservices » n'est pas un objectif. Le piège est de payer tout le coût distribué sans le bénéfice — et de s'en rendre compte au premier test de bout en bout.

🟨 Confondre base logique et instance. Une instance PostgreSQL partagée en développement est saine ; deux services qui écrivent dans la même base logique ne sont plus deux services.

🟨 Prendre le lab pour la production. Le dépôt de référence assume des choix de démonstration (vous en verrez un au chapitre suivant sur le routage). Reproduire un choix de lab sans le requalifier est un piège de ce module.

Validation des acquis

Q1. Lequel n'est pas un attribut définitoire d'un microservice ?

  • A. Persistance dédiée
  • B. Responsabilité métier unique
  • C. Partage de la base d'un autre service en écriture
  • D. Déploiement autonome
Réponse

Réponse : C. Un microservice gère sa propre persistance ; écrire dans la base d'un autre service recrée un couplage fort et supprime l'autonomie de déploiement.

Q2. Dans le socle SemiShop, où vivent aujourd'hui le panier et le paiement simulé ?

  • A. Dans deux microservices dédiés
  • B. Dans le service commandes, en modules
  • C. Dans le BFF
  • D. Dans le front React
Réponse

Réponse : B. Le socle assume un découpage minimal : le service commandes regroupe panier, statuts et paiement simulé. Les extraire est une décision à instruire, pas un acquis.

Q3. Quel coût est systématiquement introduit par une frontière de service ?

  • A. Un appel réseau faillible (latence, pannes partielles)
  • B. Une baisse de la qualité du code
  • C. L'obligation d'utiliser Kubernetes
  • D. La perte de l'historique Git
Réponse

Réponse : A. Chaque frontière remplace un appel in-process par un appel réseau : latence, sérialisation, timeouts, pannes partielles. Les autres coûts (cohérence, infra, tests) s'y ajoutent.

Q4. L'IA vous propose d'extraire un « service grade produit » qui stocke A/B/C. Bonne réaction ?

  • A. Accepter : c'est une responsabilité identifiable
  • B. Refuser : aucun signal (équipe, scaling, cycle, conformité, techno) ne le justifie
  • C. Accepter mais en Python
  • D. Demander un deuxième avis à un autre modèle
Réponse

Réponse : B. Un attribut de données n'est pas un domaine. Sans signal réel, c'est le quadrant fort coût / faible bénéfice : le microservice artificiel type.
