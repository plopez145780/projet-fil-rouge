1. L'architecture polyglotte : ce que c'est, à quoi ça sert, un exemple

Ce que c'est. Une architecture polyglotte fait coexister plusieurs langages et frameworks dans une même plateforme, un service restant mono-langage. Chaque service choisit sa stack ; le contrat d'API les fait tenir ensemble.

À quoi ça sert. Utiliser le meilleur écosystème pour chaque besoin, et surtout laisser chaque équipe travailler dans la technologie qu'elle maîtrise — c'est le cas de votre promotion, où profils Node, Python, Java et .NET cohabitent.

Un exemple simple. L'overlay polyglotte de SemiShop ajoute au socle Spring Boot quatre services : le panier en NestJS, le stock en FastAPI, les clients et le paiement en .NET 8. Le front et le BFF ne voient aucune différence : ils consomment des contrats.

2. Forces et coûts de chaque stack

Aucune de ces quatre stacks n'est « la meilleure ». Chacune a un terrain où elle excelle et un prix d'entrée :

Stack Forces Coûts / limites Terrain naturel sur SemiShop
Node.js / NestJS TypeScript bout en bout avec le front, architecture modulaire (DI, décorateurs), écosystème npm Typage à discipline (runtime JS), CPU-bound médiocre, qualité npm inégale Panier : I/O intensif, proche du front
Python / FastAPI Concision, validation Pydantic, OpenAPI natif, passerelle vers l'écosystème data/ML Performances brutes moyennes, packaging/déploiement moins normé Stock : logique simple ; futur scoring d'état du matériel
Java / Spring Boot (ou Quarkus) Robustesse transactionnelle, outillage mûr, typage fort, viviers de développeurs Verbosité, empreinte mémoire, démarrage (que Quarkus réduit nettement) Socle : commandes, catalogue, BFF — le cœur transactionnel
.NET 8 / ASP.NET Core Performances excellentes, typage fort, framework cohérent (EF Core, DI intégrée) Écosystème plus centralisé, moins répandu dans certaines DSI Clients, paiement : domaines cadrés, forte cohérence

Pour les profils techniques : le lab épingle Spring Boot 3.4.1 et .NET 8 ; Quarkus est une variante acceptée côté Java pour votre propre service — même contrat, démarrage plus rapide, expérience dev orientée conteneurs. Ce qui compte dans ce module n'est pas le classement des frameworks mais la capacité à justifier le vôtre.

3. Quand le polyglotte est justifié — et ce qu'il coûte

Les mêmes réflexes que pour le découpage : un langage de plus est une frontière de plus, avec ses signaux et ses coûts.

Trois signaux le justifient. L'équipe : les compétences existantes pèsent plus lourd que les benchmarks — une équipe .NET productive bat une équipe qui découvre Go. L'écosystème : certains besoins ont un habitat naturel (du scoring ou du RAG se prototypent en Python, difficile de faire mieux ailleurs). Le composant critique : un besoin de performance ou de plateforme qu'une stack couvre nettement mieux.

En face, les coûts sont structurels et souvent sous-estimés :

  • CI/CD multiplié : quatre toolchains, quatre familles d'images Docker, quatre stratégies de cache — vous le vivrez au module 09.
  • Standards à répliquer : conventions de logs, format d'erreurs, healthchecks, traces — tout ce qui est transverse doit exister quatre fois, à l'identique.
  • Revue croisée affaiblie : qui relit le service FastAPI si un seul membre connaît Python ? Le bus factor tombe à un.
  • Dérive des pratiques : chaque écosystème a ses idiomes ; sans conventions écrites, quatre services deviennent quatre projets étrangers.

🟨 Warning — le polyglotte de CV. Ajouter un langage « pour apprendre » ou « parce que le benchmark dit que c'est plus rapide » est le pendant linguistique du microservice artificiel. En formation, le polyglotte de SemiShop est assumé et encadré : chaque stagiaire implémente dans sa techno de prédilection — c'est le signal « équipe » à l'échelle de la promotion. En entreprise, la même décision exigerait un signal équivalent.

4. La cohérence par les contrats — et par l'IA outillée

Ce qui empêche quatre stacks de diverger, ce n'est pas la bonne volonté, ce sont deux disciplines.

Les contrats OpenAPI d'abord. Chaque service publie son contrat ; les consommateurs ne dépendent que de lui. Un développeur NestJS intègre le catalogue Java sans lire une ligne de Java. Les conventions transverses se fixent au niveau du contrat : format d'erreur commun, pagination, nommage des champs (le lab a tranché : snake_case dans les payloads), codes HTTP.

Les rules de votre ia-toolbox ensuite. Vos rules du module 02 prennent ici tout leur sens : une rule d'architecture (« un service n'accède jamais à la base d'un autre »), une rule de style par langage, une rule de contrat (« tout endpoint nouveau est décrit dans le contrat avant d'être codé »). L'IA appliquée avec ces rules produit du NestJS idiomatique et conforme aux conventions SemiShop — sans elles, elle produit du code plausible dans le style dominant de son corpus d'entraînement.

🔹 Bonne pratique production : l'IA est un excellent traducteur d'idiomes entre stacks. Montrez-lui le service catalogue Java et demandez « implémente le même endpoint healthcheck, même sémantique, dans les conventions NestJS » : c'est le moyen le plus sûr de répliquer un standard transverse dans quatre langages. La comparaison des quatre implémentations en revue est un exercice de la journée 8.

Ce qu'il faut retenir

  • 🔷 Polyglotte = un langage par service, jamais dans un service ; le contrat OpenAPI est la seule dépendance entre eux.
  • 🔷 Trois signaux justifient un langage de plus : compétences de l'équipe, écosystème nettement supérieur pour le besoin, composant critique. Le reste est du polyglotte de CV.
  • 🔷 Les coûts sont structurels : CI multipliée, standards à répliquer, revue croisée affaiblie. Ils se paient chaque semaine, pas une fois.
  • 🔷 La cohérence vient des contrats + des rules versionnées, pas de la mémoire des développeurs.

Pièges courants

🟨 Choisir la stack sur benchmark. Les écarts de débit brut entre ces quatre frameworks sont sans effet sur une boutique dont PostgreSQL est le facteur limitant. Le signal « équipe » domine presque toujours.

🟨 Laisser chaque service définir son format d'erreur. Quatre formats d'erreurs = un BFF rempli de if d'adaptation. Le format se fige dans les contrats dès maintenant, pas au module 08.

🟨 Faire confiance à l'IA pour les idiomes transverses. Sans rule explicite, l'agent générera un healthcheck différent par stack (chemin, format, codes). Tout standard transverse doit être écrit quelque part que l'IA lit.

Validation des acquis

Q1. Quel élément rend une architecture polyglotte tenable au quotidien ?

  • A. Un lead développeur qui connaît les quatre langages
  • B. Des contrats OpenAPI committés et des conventions transverses écrites
  • C. L'utilisation exclusive de frameworks récents
  • D. Un monorepo
Réponse

Réponse : B. Les services ne partagent que leurs contrats ; les conventions écrites (erreurs, logs, healthchecks) empêchent la dérive. Aucune mémoire individuelle ne tient à quatre stacks.

Q2. Quel signal justifie le plus solidement d'écrire le futur scoring d'état du matériel en Python ?

  • A. Python est plus lisible
  • B. L'écosystème data/ML de Python est nettement supérieur pour ce besoin
  • C. FastAPI génère OpenAPI nativement
  • D. Le service sera plus rapide
Réponse

Réponse : B. C'est le signal « écosystème » : le besoin a un habitat naturel. La lisibilité ou la génération OpenAPI sont des conforts, pas des justifications de frontière.

Q3. Quel coût du polyglotte se manifeste dans la CI/CD ?

  • A. Aucun, Docker uniformise tout
  • B. Une toolchain de build, de test et d'image par langage à maintenir
  • C. L'impossibilité d'utiliser GitHub Actions
  • D. Des conteneurs plus lents
Réponse

Réponse : B. Docker uniformise l'exécution, pas la construction : caches npm/pip/Maven/NuGet, versions de SDK, linters et scanners restent quadruplés. C'est l'objet du module 09.

Q4. Sans rule explicite, que produira un agent IA à qui l'on demande un healthcheck dans chaque stack ?

  • A. Quatre implémentations identiques
  • B. Quatre variantes plausibles mais divergentes (chemin, format, codes)
  • C. Une erreur
  • D. Un healthcheck uniquement en Java
Réponse

Réponse : B. L'IA suit les idiomes dominants de chaque écosystème. La convergence transverse doit être spécifiée (rule, contrat, exemple de référence) — elle ne s'improvise pas.
