Le paysage 2026 : une même boucle, des habillages différents

Tous les agents de cette page appliquent la boucle du chapitre 02 : contexte, plan, édition, commandes sous approbation, diff à valider. Ils diffèrent sur l'intégration (IDE, terminal, plateforme), la configuration projet et la souveraineté des données. Un outil, c'est un habillage de la boucle — la méthode que vous avez apprise reste la même partout.

NéoGen et la souveraineté : changer la destination, garder l'outillage

Le point décisif pour le code métier confidentiel : Cline et Continue.dev acceptent une URL d'API personnalisée. « Compatible OpenAI » (module 01) signifie que NéoGen expose la même forme d'API que les fournisseurs cloud — on garde donc exactement le même outillage (l'extension, les rules, la boucle, la checklist) et on change la destination : le code part vers l'infrastructure interne Néosoft au lieu d'un cloud tiers.

🟨 Warning : les modèles exacts servis par NéoGen et leurs capacités en mode agent sont 🟣 à vérifier dans la documentation interne Néosoft avant d'annoncer un niveau de service. À défaut de NéoGen, un modèle local (Ollama) branché sur Cline reproduit la démarche en reproductible.

Le critère souveraineté passe en premier : il élimine des options avant même de comparer le confort. Ensuite viennent l'écosystème (licences déjà en place, IDE imposé) et la préférence de travail (IDE ou terminal). Il n'y a pas de « meilleur agent » dans l'absolu — il y a un meilleur agent pour un contexte, et la compétence transférable est votre méthode de pilotage, pas la maîtrise des menus d'un produit.

Piloter en contexte polyglotte : le cas SemiShop

La salle travaille en quatre langages — NestJS, FastAPI, Spring Boot, .NET — et c'est voulu : SemiShop devient polyglotte à partir du module 04. Trois conséquences pratiques pour le pilotage d'agent :

1. La boucle de feedback est la commande de test de VOTRE stack. Un agent est aussi bon que ce qui lui permet de se vérifier. Donnez-lui, dans le prompt ou les rules, la commande de vérification de votre service — c'est elle qui ferme la boucle verify :

Stack Commande de vérification
Spring Boot (Gradle) ./gradlew test
NestJS npm test (Jest)
FastAPI pytest
.NET dotnet test
Quarkus ./mvnw test

2. Les rules se déclinent par langage, les critères non. ia-toolbox/rules/ gagne un fichier par stack (conventions de nommage, ORM retenu, structure de dossiers) ; mais la checklist de validation du diff, les garde-fous et les critères de livrable (bout-en-bout, pas de dépendance inventée, historique lisible) sont identiques pour tous. C'est ce qui rend les revues croisées possibles entre stagiaires de stacks différentes.

3. Le contrat d'API est la lingua franca. Au TP 3.1, chacun implémente le catalogue dans sa techno, mais tous produisent le même contrat (routes, pagination, codes d'erreur). Un agent piloté sur FastAPI et un agent piloté sur Spring Boot doivent converger vers des réponses JSON indistinguables — c'est le contrat qu'on valide, pas le langage. Cette discipline prépare directement les contrats OpenAPI du module 04 et les tests contractuels du module 08.

Pour les profils techniques : les agents s'appuient sur le LSP et l'outillage du langage pour naviguer et se corriger. Une stack au tooling scriptable et rapide (tests en secondes, build reproductible) donne un agent nettement plus efficace — encore un argument pour des services petits et des tests rapides, quel que soit le langage.

Ce qu'il faut retenir

  • Tous les outils appliquent la même boucle ; ils diffèrent par l'intégration, la configuration projet et la souveraineté.
  • Souveraineté d'abord : code sensible → NéoGen (ou local) via Cline / Continue.dev, grâce à la compatibilité OpenAI.
  • En polyglotte : rules par stack, critères communs, contrat d'API partagé — et chaque agent reçoit la commande de test de sa stack.
  • La compétence transférable est la méthode de pilotage, pas la maîtrise d'un produit.

Pièges courants

🟨 Choisir l'outil avant le contexte : la question n'est pas « quel est le meilleur », mais « lequel est admissible ici (souveraineté), puis lequel s'intègre le mieux ».
🟨 Piloter toutes les stacks avec les mêmes rules techniques : les conventions NestJS ne sont pas celles de Spring ; l'agent générera des hybrides.
🟨 Suivre un tutoriel daté : les noms de modes bougent vite (Copilot Edits, Composer) ; se référer à la documentation officielle du moment.

Bonnes pratiques production

🔹 Documenter le choix d'outil dans docs/decisions.md (module 02) : contexte, critère décisif, alternatives écartées — comme toute décision d'architecture.
🔹 Un fichier de rules par stack dans ia-toolbox/rules/, plus un fichier commun (garde-fous, checklist) : la boîte à outils reste utile quel que soit le service.
🔹 Tester la bascule NéoGen tôt : configurer une fois Cline ou Continue.dev sur l'API interne, pour que le jour où le code devient sensible, la bascule soit un changement d'URL, pas un changement d'outillage.

Validation des acquis

Q1. Vous manipulez du code métier confidentiel et voulez un mode agent. Quel choix est cohérent avec la souveraineté ?

  • A. N'importe quel agent cloud, c'est équivalent
  • B. NéoGen (interne), branché via Cline ou Continue.dev sur son API compatible OpenAI
  • C. Copier le code sur un service public pour aller plus vite
  • D. Désactiver tout assistant
Réponse

Réponse : B. La compatibilité OpenAI de NéoGen permet de garder l'outillage (extension, rules, boucle) en changeant la destination : le code reste dans l'infrastructure interne.

Q2. Qu'est-ce qui distingue fondamentalement NéoGen des autres lignes du comparatif ?

  • A. Il est plus rapide
  • B. Ce n'est pas un agent mais un backend de modèles, qu'on branche sur un client agent
  • C. Il ne fonctionne qu'avec Java
  • D. Il n'a pas d'API
Réponse

Réponse : B. NéoGen fournit les modèles (vLLM, API compatible OpenAI) ; la boucle agent est apportée par le client (Cline, Continue.dev) qui pointe dessus.

Q3. En contexte polyglotte, qu'est-ce qui doit rester identique entre les stacks ?

  • A. Le langage de programmation
  • B. Les conventions de nommage de chaque framework
  • C. Les critères de validation (checklist du diff, contrat d'API, livrables)
  • D. La commande de lancement des tests
Réponse

Réponse : C. Les rules techniques se déclinent par stack, la commande de test aussi ; mais la checklist, les garde-fous et le contrat d'API sont communs — c'est ce qui rend les services interopérables et les revues croisées possibles.

Q4. Pourquoi donner à l'agent la commande de test de votre stack dès le prompt ou les rules ?

  • A. Pour qu'il aille plus vite à compiler
  • B. Parce que c'est elle qui ferme la boucle verify : sans vérification exécutable, l'agent ne peut pas se corriger
  • C. Parce que l'agent refuse de démarrer sans
  • D. Pour éviter d'écrire des tests soi-même
Réponse

Réponse : B. Un agent est aussi bon que sa boucle de feedback. La commande de test est le mécanisme de vérification qui rend le verify objectif.
