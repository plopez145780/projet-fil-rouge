Objectifs pédagogiques

À la fin de ce chapitre, vous saurez :

  • Décrire la boucle agent plan-act-verify et situer les points où le développeur reprend la main.
  • Justifier pourquoi la validation du plan est le contrôle le plus rentable d'un tour de boucle.
  • Dérouler la séquence de validation d'un diff : périmètre, dépendances, secrets, tests, relecture — dans cet ordre.
  • Décider quand accepter, redécouper ou renvoyer au plan.

Une boucle, pas un jet unique

Un agent ne produit pas du code d'un bloc. Il tourne dans une boucle en trois temps, que vous pilotez :

  1. Plan — l'agent transforme votre intention en étapes vérifiables : quels fichiers, dans quel ordre, quels tests. Vous validez le plan avant qu'il touche au code. C'est le point de contrôle le plus rentable : corriger un plan coûte une phrase, corriger 400 lignes générées coûte une heure.
  2. Act — l'agent édite les fichiers et, s'il en a le droit, lance des commandes. Il travaille par petits incréments, pas en un patch géant.
  3. Verify — l'agent et vous vérifiez : les tests passent-ils, le diff est-il conforme au plan, rien n'a débordé ? Si un écart apparaît, on repart au plan — on ne rafistole pas au jugé.

Concrètement, sur SemiShop : vous demandez « ajoute GET /produits/{id}/disponibilite au service catalogue ». Un agent bien piloté répond d'abord par un plan — créer le DTO, ajouter la méthode de service avec le cas 404, ajouter la route, écrire un test, lancer ./gradlew test. Vous lisez ces cinq lignes, vous corrigez si besoin (« le test d'abord »), puis seulement vous le laissez écrire. Au TD 3.1, vous disséquerez cette trace précise.

🔷 Les noms varient, la boucle reste. Chaque outil a son vocabulaire : Cline sépare explicitement les modes Plan et Act ; Claude Code et Cursor proposent un plan mode en lecture seule avant l'édition ; Anthropic décrit la boucle de ses agents comme « rassembler le contexte → agir → vérifier le travail → recommencer ». plan-act-verify est notre modèle pédagogique commun — il se projette sur n'importe quel outil, y compris ceux qui ne nomment pas leurs phases.

🟨 Warning : le piège classique est de sauter l'étape plan. On tape l'objectif, l'agent part, produit un gros diff — et on se retrouve à relire à froid du code qu'on n'a pas pensé. Faites toujours expliciter le plan d'abord ; si l'outil a un mode plan, utilisez-le.

Le diff : le livrable, donc le point de validation

Le diff — l'ensemble des lignes ajoutées et supprimées — est ce que l'agent vous livre à chaque tour. C'est là que se joue la validation humaine. On n'accepte jamais un diff « parce que les tests sont verts » : des tests peuvent être faux, tautologiques ou absents. On déroule une séquence de contrôles, dans cet ordre :

    t_human -->|"un controle echoue"| t_reject["Rejet - retour au plan"]
  • 1. Périmètre : l'agent n'a touché que ce qui était prévu ? Un application.yaml modifié « au passage », un renommage sauvage = signal d'alarme.
  • 2. Dépendances : toute librairie ajoutée existe réellement (les modèles hallucinent des packages aux noms plausibles — spring-boot-starter-pagination n'existe pas), est à jour, et surtout est nécessaire. Une dépendance inutile alourdit la surface d'attaque.
  • 3. Secrets : aucun mot de passe, token ou URL interne en clair dans le diff — y compris dans les fichiers de configuration, pas seulement le code.
  • 4. Tests : verts et significatifs (pas un assertTrue(true)). Et « Build OK » n'est pas « tests verts ». On y reviendra en profondeur au module 08.
  • 5. Relecture humaine : vous lisez le diff ligne à ligne. Non négociable — c'est la validation humaine du triptyque.

L'ordre a un sens : les deux premiers contrôles se font en quelques secondes sur la liste des fichiers et des dépendances, avant même de lire le code. Si le périmètre a débordé ou qu'un package est douteux, inutile d'aller plus loin — retour au plan.

🟥 Erreur : critique — accepter un gros patch non relu « parce que ça compile ». Un diff qu'on n'a pas lu est un diff qu'on ne maîtrise pas. S'il est trop gros pour être relu, le périmètre était trop large : redécoupez en incréments plus petits, ne vous forcez pas à relire l'illisible.

Capture officielle de la documentation VS Code (code.visualstudio.com, 2026) : le point de validation humaine - chaque fichier modifie par l agent s accepte ou se rejette explicitement.

Trois issues possibles, jamais quatre

Chaque tour de verify se conclut par une décision explicite :

Verdict Quand Geste
🟢 Accepter Les 5 contrôles passent Commit de l'incrément, message clair
🟨 Redécouper Diff conforme mais trop gros pour être relu sérieusement Rejeter, relancer sur un périmètre plus étroit
🟥 Retour au plan Un contrôle échoue (débordement, dépendance douteuse, secret...) Rejeter le diff, corriger le plan, relancer

La quatrième issue — « accepter en se promettant de relire plus tard » — n'existe pas. Plus tard, le diff suivant sera empilé dessus.

Ce qu'il faut retenir

  • La boucle plan-act-verify : valider le plan avant le code ; en cas d'écart, revenir au plan, pas rafistoler.
  • Le diff est le livrable : périmètre → dépendances → secrets → tests → relecture, dans cet ordre.
  • « Tests verts » n'est pas une preuve suffisante ; « Build OK » encore moins.
  • Trois issues : accepter, redécouper, retour au plan. Jamais « accepter sans relire ».

Pièges courants

🟨 Sauter l'étape plan : on subit ensuite un diff qu'on n'a pas pensé.
🟨 Rafistoler en verify : corriger à la main un diff non conforme, au lieu de corriger le plan — l'erreur de fond se reproduira au tour suivant.
🟨 Croire le compteur de tests : 24 tests verts ne disent rien de la qualité des 24 assertions.

Bonnes pratiques production

🔹 Un incrément = un commit : le diff validé part immédiatement en commit ; le tour suivant démarre sur une base propre.
🔹 Coder la séquence dans les rules (ia-toolbox/rules/) : « plan explicite avant toute modification multi-fichiers », « aucune dépendance sans justification écrite ». L'agent lit ces rules et s'y conforme — le TD 3.1 en fera une checklist versionnée.
🔹 Contexte compact (module 02) : moins de tokens, moins de latence, moins de dérive dans les longs tours de boucle.

Validation des acquis

Q1. Dans la boucle plan-act-verify, quand reprend-on la main de la façon la plus rentable ?

  • A. Après le commit
  • B. En validant le plan avant toute écriture de code
  • C. Uniquement si le build casse
  • D. Jamais, l'agent est autonome
Réponse

Réponse : B. Corriger un plan coûte une phrase ; corriger des centaines de lignes générées coûte une heure. Le contrôle le plus rentable est en amont.

Q2. Quel est l'ordre correct de la séquence de validation d'un diff ?

  • A. Tests → relecture → périmètre → dépendances → secrets
  • B. Périmètre → dépendances → secrets → tests → relecture ligne à ligne
  • C. Relecture → tests → secrets → périmètre → dépendances
  • D. Secrets → tests → relecture → périmètre → dépendances
Réponse

Réponse : B. On vérifie d'abord ce qui se voit sans lire le code (fichiers touchés, dépendances), puis les secrets, les tests, et enfin la relecture complète.

Q3. L'agent présente un diff de 600 lignes sur 12 fichiers pour une tâche que vous pensiez petite. Bonne réaction ?

  • A. L'accepter si les tests passent
  • B. Le rejeter et redécouper le périmètre en incréments plus petits
  • C. Committer d'abord, relire ensuite
  • D. Supprimer les tests pour aller plus vite
Réponse

Réponse : B. Un diff trop gros pour être relu ligne à ligne est un diff non maîtrisé. Le signal correct est de redécouper.

Q4. En verify, vous constatez que l'agent a modifié un fichier hors périmètre. Vous corrigez ce fichier à la main et acceptez le reste. Quel est le problème ?

  • A. Aucun, c'est pragmatique
  • B. On a rafistolé au lieu de corriger le plan : la cause du débordement se reproduira au tour suivant
  • C. Il fallait committer le fichier hors périmètre séparément
  • D. Il fallait relancer les tests deux fois
Réponse

Réponse : B. Le débordement vient d'un plan ou d'un cadrage défaillant. On rejette, on corrige le plan, on relance — sinon l'écart devient une habitude coûteuse.
