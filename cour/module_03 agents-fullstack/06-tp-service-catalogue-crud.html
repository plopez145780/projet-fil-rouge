Objectif et contexte

Objectif. Faire construire par un agent, en petites étapes validées, le premier service de SemiShop : le catalogue (référentiel produits). Vous ne tapez presque pas de code — vous planifiez, pilotez, et validez chaque diff avec votre checklist. À la fin : un flux prouvé front → Traefik → catalogue → PostgreSQL.

Contexte fil rouge. SEMIFIR vend du matériel reconditionné : chaque produit porte un grade (A : excellent état, B : bon état, C : état correct), un prix HT, un stock. Le catalogue est le socle de toute l'application — les services commandes, panier et le BFF construits dans les modules suivants s'appuieront dessus. Jusqu'ici votre dépôt contient l'infrastructure (compose v0) et la boîte à outils IA ; aujourd'hui, le premier code métier.

🔷 Langage au choix. La variante de référence est Spring Boot 3.4.1 / Java 21 (celle du lab de correction). Les variantes NestJS, FastAPI, .NET 8 et Quarkus sont acceptées : les critères de validation sont identiques quel que soit le langage. Le contrat d'API et le modèle de données ci-dessous sont votre cahier des charges, pas le code d'une stack.

Contrat d'API (le service écoute sur le port 8081, sans préfixe /api — c'est Traefik qui le retire) :

Verbe et route Comportement Réponse

Exigences transverses : schéma créé par migrations versionnées (Flyway, Alembic, EF Core migrations... selon la stack — jamais un ddl-auto/synchronize automatique) ; seed minimal (au moins 2 catégories, 4 produits avec des grades variés) ; taille de page bornée côté serveur ; aucun secret en dur (identifiants de base lus depuis l'environnement).

Étape 1 : cadrer et faire produire le plan

Objectif. Écrire l'intention et obtenir de l'agent un plan en incréments vérifiables — avant toute ligne de code.

Contexte. C'est le tour de boucle le plus rentable. Un plan clair rend chaque diff suivant petit et relisible. Le périmètre du TP est volontairement découpé en 4 incréments : schéma + entités, API CRUD, exposition Traefik, front.

Explication. On donne à l'agent le cahier des charges ci-dessus, la stack choisie, les rules (ia-toolbox/rules/) et les contraintes d'environnement (base referentiel_db existante, service sur 8081). On lui demande un plan d'abord, pas du code.

Exemple traité — un prompt de cadrage (variante Spring Boot, à adapter à votre stack) :

Contexte : depot semishop, compose v0 demarre (Traefik :80, PostgreSQL 16 avec la base
referentiel_db, user semishop). Stack : Java 21 / Spring Boot 3.4.1 / Gradle.
Objectif : service catalogue dans services/catalogue/ — referentiel produits
reconditionnes. Tables categorie et produit (grade A/B/C, prix_ht, stock, sku unique).
API REST sur :8081 : GET /produits pagine et filtre (categorie, q, grade, page, taille),
GET /produits/{id}, POST, PUT, DELETE, GET /categories.
Contraintes : migrations Flyway (pas de ddl-auto create), seed minimal (2 categories,
4 produits), taille de page bornee cote serveur, identifiants de base lus depuis
l'environnement, respecter ia-toolbox/rules/, aucune dependance non justifiee.
Ne code rien encore : propose d'abord un PLAN en 4 increments verifiables
(1 schema+entites, 2 API CRUD, 3 conteneur+Traefik, 4 front), un commit par increment.

À vous !

Rédigez votre prompt de cadrage (adapté à votre stack) et faites produire le plan. Validez-le avant de continuer : chaque incrément doit être testable seul (une commande de vérification par incrément) et dans le bon ordre. Corrigez le plan si un incrément est trop gros — c'est une phrase maintenant, une heure plus tard.

Erreurs fréquentes à cette étape :

Plan en un seul bloc « faire le service » Périmètre pas décomposé Aucune étape intermédiaire vérifiable Imposer les 4 incréments

Étape 2 : incrément 1 — schéma, migrations, entités

Objectif. Faire produire les migrations versionnées (schéma + seed), les entités et la connexion à referentiel_db, puis valider le diff et prouver que les tables existent.

Explication. La migration est le contrat avec la base : elle se relit comme du code. Deux fichiers : V1 crée categorie et produit avec leurs contraintes (CHECK sur le grade, UNIQUE sur le sku), V2 insère le seed. L'application, elle, ne fait que valider le mapping au démarrage — le schéma appartient aux migrations.

Exemple traité — l'amorce de la première migration (variante Spring Boot / Flyway ; l'agent complète la table produit d'après le cahier des charges) :

-- services/catalogue/src/main/resources/db/migration/V1__schema.sql
-- AMORCE : table categorie complete ; table produit a faire produire par l'agent.

CREATE TABLE categorie (
    id        BIGSERIAL    PRIMARY KEY,
    libelle   VARCHAR(120) NOT NULL,
    slug      VARCHAR(140) NOT NULL UNIQUE,
    parent_id BIGINT       REFERENCES categorie(id)
);

-- A faire produire : CREATE TABLE produit (...) avec les contraintes du cahier
-- des charges : grade IN ('A','B','C'), prix_ht >= 0, stock >= 0, sku UNIQUE.

🔷 Les identifiants de connexion se lisent depuis l'environnement avec un défaut de dev, jamais une valeur de production en dur — par exemple url: ${DB_URL:jdbc:postgresql://localhost:5432/referentiel_db} côté Spring, ou l'équivalent .env de votre stack.

À vous !

Pilotez l'agent pour produire les migrations (V1 schéma, V2 seed : au moins 2 catégories et 4 produits reconditionnés avec des grades variés), les entités/modèles et la configuration de connexion. Appliquez votre checklist au diff : périmètre (rien d'autre que le service), dépendances (le strict nécessaire : web, ORM, driver PostgreSQL, moteur de migration), secrets (identifiants lus depuis l'environnement). Puis démarrez le service et prouvez que le schéma est en base :

docker compose exec postgres psql -U semishop -d referentiel_db -c "\dt"
docker compose exec postgres psql -U semishop -d referentiel_db \
  -c "SELECT sku, grade, prix_ht, stock FROM produit;"

Committez l'incrément (ex. feat(catalogue): schema, migrations Flyway et entites).

Erreurs fréquentes à cette étape :

🟨 Symptôme Cause probable Comment le détecter Correction
password authentication failed Mauvais utilisateur / variable non lue Le service ne démarre pas Vérifier user semishop, base referentiel_db, variables d'env
L'agent choisit ddl-auto: create au lieu des migrations Facilité par défaut Pas de dossier db/migration/ dans le diff Exiger les migrations ; l'appli valide, elle ne crée pas
Le seed passe par le code applicatif Confusion seed / logique métier Des INSERT dans une classe Java ou un service Déplacer le seed dans V2__seed.sql

Étape 3 : incrément 2 — CRUD REST paginé et recherche

Objectif. Exposer le contrat d'API complet, avec pagination bornée et filtres combinables, puis valider le diff et tester à la main.

Explication. Le point délicat n'est pas le CRUD, c'est la liste : trois filtres facultatifs (categorie, q, grade) qui se combinent, une pagination dont la taille est bornée côté serveur (un client qui demande taille=100000 doit recevoir la borne max, pas mettre la base à genoux), et une réponse qui expose le total (totalElements) pour que le front pagine. Les valeurs de grade se valident (A/B/C, sinon 400) — jamais de valeur cliente injectée telle quelle dans une requête.

Exemple traité — vérification de la liste filtrée (la forme attendue, quel que soit le langage) :

# grade A uniquement, page de 2
curl -s "http://localhost:8081/produits?grade=A&page=0&taille=2"

{"content":[{"id":1,"sku":"NS-LAP-0001","libelle":"...","grade":"A","prixHt":549.00,
"stock":5,...},{...}],"page":0,"taille":2,"totalElements":4}

À vous !

Faites produire le CRUD complet par l'agent, un sous-incrément à la fois si le diff dépasse ce que vous pouvez relire (liste d'abord, écritures ensuite). Validez chaque diff avec la checklist, puis testez :

curl -s "http://localhost:8081/produits?q=reconditionne&taille=5"     # recherche texte
curl -s "http://localhost:8081/produits?grade=Z"                       # attendu : 400
curl -s "http://localhost:8081/produits/999"                           # attendu : 404
curl -s -X POST http://localhost:8081/produits \
  -H "Content-Type: application/json" \
  -d '{"sku":"NS-PER-0099","libelle":"Webcam Logitech C920 reconditionnee",
       "categorieId":2,"grade":"B","prixHt":39.00,"stock":6}'          # attendu : 201

Rejouez le POST à l'identique : attendu 409 (sku déjà pris), pas une erreur 500. Committez (ex. feat(catalogue): CRUD REST pagine et recherche).

Erreurs fréquentes à cette étape :

🟨 Symptôme Cause probable Comment le détecter Correction
taille=100000 accepté Pagination non bornée La requête renvoie tout Borner côté serveur (défaut 20, max 100 par exemple)
grade=Z renvoie 200 et une liste vide Valeur non validée Pas de 400 Valider A/B/C, refuser le reste
Doublon de sku renvoie 500 Contrainte base non traduite Stacktrace au lieu d'un 409 Contrôler l'existence avant insertion, renvoyer 409
L'entité JPA/ORM exposée brute en écriture Pas de DTO / schéma d'entrée id ou date_ajout acceptés dans le POST DTO d'entrée distinct de l'entité

Étape 4 : incrément 3 — conteneuriser et exposer via Traefik

Objectif. Mettre le service dans le compose et le rendre joignable sur http://localhost/api/produits, comme les autres services le seront.

Explication. Traefik route par labels déclarés sur le service Docker : une règle d'entrée (PathPrefix), un middleware stripprefix qui retire /api avant de transmettre (le service, lui, écoute sur /produits sans préfixe), et le port interne. C'est l'état M03 : le catalogue est exposé en direct ; à partir du module 04, un BFF prendra la place derrière /api et le catalogue redeviendra interne.

Exemple traité — le bloc à ajouter au docker-compose.yml (c'est de l'infrastructure, pas la solution du CRUD — vous pouvez le donner tel quel à l'agent) :

  catalogue:
    build:
      context: ./services/catalogue
    environment:
      DB_URL: "jdbc:postgresql://postgres:5432/referentiel_db"   # adapter a votre stack
      DB_USER: "${POSTGRES_USER:-semishop}"
      DB_PASSWORD: "${POSTGRES_PASSWORD:-semishop}"
    depends_on:
      postgres:
        condition: service_healthy
    labels:

      - "traefik.enable=true"
      - "traefik.http.routers.catalogue.rule=PathPrefix(`/api`)"
      - "traefik.http.routers.catalogue.entrypoints=web"
      - "traefik.http.middlewares.api-strip.stripprefix.prefixes=/api"
      - "traefik.http.routers.catalogue.middlewares=api-strip"
      - "traefik.http.services.catalogue.loadbalancer.server.port=8081"
    networks:

      - semishop

À vous !

Faites produire le Dockerfile du service par l'agent (build multi-stage : compilation puis image d'exécution légère), ajoutez le bloc compose, puis :

docker compose up -d --build catalogue
docker compose ps                                  # catalogue "running"
curl -s "http://localhost/api/produits?taille=3"   # via Traefik cette fois

La réponse doit être identique à celle du port 8081 en direct. Relisez le diff du Dockerfile (image de base épinglée en version précise, pas de latest ; aucun secret copié dans l'image). Committez (ex. feat(catalogue): Dockerfile et exposition Traefik).

Erreurs fréquentes à cette étape :

🟨 Symptôme Cause probable Comment le détecter Correction
404 via Traefik, 200 en direct Stripprefix absent : le service reçoit /api/produits Comparer les deux URLs Ajouter le middleware stripprefix
Connection refused du service vers la base URL avec localhost dans le conteneur Logs du conteneur catalogue Dans le compose, l'hôte base est postgres, pas localhost
Image qui embarque le JDK complet / node_modules de dev Pas de multi-stage Image de plusieurs Go Build multi-stage, image d'exécution minimale
Tag latest sur l'image de base Facilité de l'agent FROM xxx:latest dans le diff Épingler une version précise et récente

Étape 5 : incrément 4 — amorce du front, la page catalogue

Objectif. Créer le front React et une page CatalogPage minimale qui liste les produits via Traefik, puis valider dans le navigateur.

Explication. La variante de référence du fil rouge est React 18 + Vite 5 (alignée sur le lab de correction ; des majeures plus récentes existent, on épingle celles du socle commun). En dev, le serveur Vite tourne sur 5173 et proxifie /api vers Traefik : le front appelle /api/produits en chemin relatif, aucun CORS à configurer, et le même code fonctionnera tel quel une fois le front servi par Traefik. La page reste volontairement minimale — libellé, grade, prix, stock — les filtres et le panier viendront dans les modules suivants.

Exemple traité — le proxy Vite et l'amorce de la page :

// frontend/vite.config.js -- proxy dev : /api -> Traefik
import { defineConfig } from "vite";
import react from "@vitejs/plugin-react";

export default defineConfig({
  plugins: [react()],
  server: {
    proxy: { "/api": "http://localhost:80" },
  },
});

// frontend/src/pages/CatalogPage.jsx -- AMORCE : a completer par l'agent
import { useEffect, useState } from "react";

export default function CatalogPage() {
  const [produits, setProduits] = useState([]);
  const [erreur, setErreur] = useState(null);

  useEffect(() => {
    fetch("/api/produits?page=0&taille=12")
      .then((r) => { if (!r.ok) throw new Error("Erreur " + r.status); return r.json(); })
      .then((data) => setProduits(data.content))
      .catch((e) => setErreur(e.message));
  }, []);

  // A completer : affichage de la grille (libelle, grade, prix HT, stock)
  // et d'un message si erreur ou liste vide.
  return <main>{/* a completer */}</main>;
}

À vous !

Créez le projet front (npm create vite@5 — gabarit React), faites compléter CatalogPage par l'agent (grille des produits : libellé, grade, prix, stock ; message d'erreur si l'API est injoignable ; message « aucun produit » si la liste est vide). Validez le diff — attention aux dépendances front : fetch natif suffit, pas de client HTTP ni de librairie UI pour ce périmètre. Puis :

cd frontend && npm install && npm run dev    # http://localhost:5173

Les produits du seed s'affichent dans le navigateur. Committez (ex. feat(front): amorce CatalogPage - liste des produits).

Étape 6 : vérification bout-en-bout, historique, doc

Objectif. Prouver le flux complet front → Traefik → catalogue → PostgreSQL, vérifier l'historique Git, documenter.

Explication. On ne suppose pas le bout-en-bout, on le prouve : une donnée créée par l'API doit apparaître dans le navigateur et dans un SELECT direct en base. Puis on relit l'historique : c'est lui qui raconte le TP.

Exemple traité — la preuve de persistance :

docker compose exec postgres psql -U semishop -d referentiel_db \
  -c "SELECT sku, libelle, grade, stock FROM produit ORDER BY id;"

     sku      |               libelle                | grade | stock
--------------+--------------------------------------+-------+-------
 NS-LAP-0001  | Dell Latitude 7420 reconditionne     | A     |     5
 ...
 NS-PER-0099  | Webcam Logitech C920 reconditionnee  | B     |     6

À vous !

  1. Créez un produit via POST (route Traefik http://localhost/api/produits), rechargez la page front : il apparaît. Vérifiez-le en base avec le SELECT ci-dessus.
  2. Rédigez docs/catalogue.md : le modèle de données, le contrat d'API (tableau), la procédure de lancement.
  3. Contrôlez l'historique :
git log --oneline    # au moins 4 commits jalonnes : schema, CRUD, Traefik, front

🔷 Point de contrôle livrable : flux bout-en-bout prouvé (navigateur + SELECT), chaque diff relu avec la checklist, aucune dépendance inventée, pagination bornée, migrations versionnées, aucun secret en dur, historique d'au moins 4 commits aux messages clairs, docs/catalogue.md présent.

Erreurs fréquentes à cette étape :

🟨 Symptôme Cause probable Comment le détecter Correction
Historique en un seul gros commit Discipline perdue en route git log montre 1-2 lignes La prochaine fois : commit à chaque incrément validé ; ne pas réécrire l'historique maintenant, le noter dans la doc

Correction / Solution :
🔷 Correction distribuée par le formateur (voir corriges/corrige-06-tp-service-catalogue-crud.md). L'implémentation de référence complète est le service catalogue du lab SemiShop — le corrigé donne les chemins précis et les extraits clés.

🔹 En production, ce que ça change : construire un service en incréments relus et commités, avec migrations versionnées et pagination bornée, c'est exactement ce qu'on attend d'une PR de qualité — que le code vienne d'un agent ou d'un humain. L'historique jalonné n'est pas de la cosmétique : c'est lui qui permet la revue, le git bisect et le retour arrière ciblé quand un bug surgit trois modules plus tard.

Récap

🟢 Ce que vous savez maintenant faire :

  • Piloter un agent pour construire un microservice complet par petits incréments validés.
  • Imposer des migrations versionnées et une pagination bornée dans le plan, pas après coup.
  • Exposer un service derrière Traefik (règle, stripprefix, port) et le prouver.
  • Prouver un flux bout-en-bout (navigateur + SELECT), pas le supposer.

🟨 Pièges à retenir :

  • Un diff trop gros pour être relu = périmètre trop large : redécouper.
  • ddl-auto/synchronize n'est pas une stratégie de schéma : migrations.
  • « Ça s'affiche » n'est pas « c'est persisté » : vérifier en base.

🔷 Points clés à retenir :

  • Le contrat d'API est le même pour toutes les stacks — c'est lui qu'on valide, pas le langage.
  • L'exposition directe via Traefik est l'état M03 ; le BFF la remplacera au module 04.
  • Environnement et code conservés pour la suite.
